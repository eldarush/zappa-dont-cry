# Zappa Harness

External validation harness for the `zappa-dont-cry` QaaS skill namespace.

Reports are written to `D:\QaaS\_tmp\zappa-dont-cry\harness-runs\<timestamp>`.

The harness is intentionally outside the skill namespace so skill folders remain one folder per unique skill.

The `environment` suite bootstraps a fresh dynamic root under the harness run directory, checks the manifest schema, verifies skill/session/promotion policy links, separates weak-model dry and live output directories, and proves bootstrap idempotence with sentinel files.

The `default-environment` suite bootstraps and checks the default long-lived work root at `D:\QaaS\_tmp\zappa-dont-cry`, so the reusable environment does not drift behind the run-local environment contract.

The `skill-output-contracts` suite checks each zappa skill for the concrete output fields weak models must produce.

The `objective-coverage` suite checks the skill pack against the original zappa objective capability map and writes `D:\QaaS\_tmp\zappa-dont-cry\coverage\objective-capability-coverage.json`. That record must stay `not_complete` while live weak-model validation and executable QaaS promotion are blocked.

The `intent-assumptions` suite requires every generated manifest assumption to include `assumption`, `why_safe`, `risk_if_wrong`, `how_to_override`, and `public_evidence`.

The `qaas-yaml-schema` suite checks generated Runner and Mocker YAML top-level keys against the public schema assets and requires placeholder content to remain blocked.

The `promotion-seed` suite regenerates `D:\QaaS\_tmp\zappa-dont-cry\generated-tests\promotion-seed\qaas-docs-hello-world-http` from public QaaS docs. It runs real `dotnet new`, package pinning, `dotnet build`, Runner/Mocker `template`, and local Runner/Mocker live validation. It still keeps the seed blocked until live airgapped weak-model validation passes.

The `promotion-packet` suite writes `D:\QaaS\_tmp\zappa-dont-cry\coverage\promotion-packet-summary.json` and only writes per-manifest packet files under `D:\QaaS\_tmp\zappa-dont-cry\promotion-packets` for executable manifests with complete template/build/live/airgapped evidence. With only blocked skeletons, it passes structurally with `promotion_packet_status: blocked` and `packet_count: 0`.

The `harness-regression` suite runs fixture-owned fail-closed regression tests outside the skills folder. It checks SeedRoot path escapes, spoofed promotion-seed YAML exceptions, dry-run weak evidence promoted as passed, artifact/transcript paths outside the seed root, manifest-declared C# compile scope, selected-candidate promotion leakage, FastAPI route/body/lifecycle/process guardrails, FastAPI build/template/live/runtime QaaS spoofing, FastAPI live QaaS stale-summary/hash/double-slash/weak-spoof guardrails, FastAPI lifecycle cleanup and evidence path binding, stale passed lifecycle summary rejection, Gin route/body/lifecycle/process guardrails, Gin lifecycle/module/weak/QaaS spoof guardrails, Gin managed-Go spoofing for bad go.mod/go.sum hashes, missing managed caches, mutable module commands, ambient Go source spoofing, wrong managed Go version, wrong official archive URL/SHA, and `go-env.json` path drift, Gin selected-live spoofing across stale summaries, fake hashes, mixed validation records, managed Go source spoofing, `go-env.json` drift, double-slash live URLs, weak-spoof, wrong-repo, failed-summary, and body-drift cases, Crawl4AI deferred-contract, healthcheck-only candidate, and Docker lifecycle guardrails, including exact-200, `/crawl`, body/schema assertion, unsafe Docker cleanup, body-reading custom-hook spoofing, weak/QaaS lifecycle spoofing, status >= 400, user-owned container names, protected container drift, and forged response hashes, Flask lifecycle blocker/hash/managed-venv spoofing, Express lifecycle weak/QaaS/body/hash/package-evidence spoofing, Flask `ExactHttpTextBody` sidecar spoofing across active YAML, fuzzy hook code, sidecar source paths, expected-body hashes, runtime-plan packets, and hook-plan packets, plus Flask selected-live spoofing across stale summaries, fake hashes, mixed validation records, missing lifecycle adoption, stale runtime hook status, missing staged `Route: ''`, double-slash live URLs, source YAML custom-assertion activation, weak-spoof, wrong-repo, failed-summary, and body-drift cases.

The `top-repo-triage` suite writes `D:\QaaS\_tmp\zappa-dont-cry\coverage\top-repo-promotion-triage.json`. It ranks the 250 harvested repository contracts into promotion-candidate review, contract-review priority, and discovery backlog groups while keeping `completion_ready: false` until selected artifacts have real QaaS template/build/live/airgapped evidence.

The `selected-contracts` suite validates exact small public GitHub blob snapshots for selected top-repo candidates. It recomputes Git blob SHA and SHA-256, verifies UTF-8 text evidence under the managed selected-contracts root, rejects selected evidence written into generated tests, and fails if selected evidence alone makes any top-repo manifest executable.

The `selected-candidates` suite validates blocked candidate packets generated from selected immutable public contracts. It currently checks repo-aware packets for `fastapi/fastapi`, `gin-gonic/gin`, `typicode/json-server`, `pallets/flask`, `expressjs/express`, and `unclecode/crawl4ai`, requires explicit deferred records for selected repos without generated packets, rejects UTF-8 BOMs on generated candidate artifacts, rejects invented QaaS lifecycle fields, keeps process startup/cleanup outside QaaS YAML, and fails if a candidate is written under `promotion-packets` or marked executable before airgapped evidence exists. The FastAPI packet uses documented `ObjectOutputJsonSchema` to enforce the exact README JSON object for `GET /items/5?q=somequery` and rejects process commands, query/body invention, route drift, lifecycle-blocker removal, partial/spoofed QaaS validation claims, and lifecycle/live adoption without real summary/transcript/response evidence bound to the current manifest and runtime plan. It accepts FastAPI QaaS evidence only when build, template, live, selected-candidate QaaS, and runtime QaaS records are all passed, the summary is adopted by the manifest/runtime plan, and hashes are recomputed from the actual source/staged files. The Gin packet uses documented `ObjectOutputJsonSchema` to enforce the exact README JSON object for `GET /ping` and rejects Go process commands in YAML, invented body fields, route/port drift, schema drift, and lifecycle-blocker removal. The checker accepts Flask QaaS evidence only when lifecycle, assertion build, runner build, template, live, selected-candidate QaaS, and runtime QaaS records are all passed, source/staged hashes are recomputed, manifest/runtime records match the live summary, and airgapped weak-model validation remains blocked. The Express packet uses the immutable README `GET /` example on port 3000 with exact `Hello World` body evidence, keeps `node app.mjs` and `npm install express` in the external runtime plan instead of QaaS YAML, includes an `ExactHttpTextBody` sidecar packet with repository-specific namespace, and remains blocked until Node lifecycle/cleanup, custom assertion build/template/live, QaaS template/live, strong review, and live airgapped weak-model evidence pass.

The `completion-readiness` suite writes `D:\QaaS\_tmp\zappa-dont-cry\blockers\objective-completion-readiness.json`, separating structural harness status from objective completion readiness.

The `strong-review` suite should be run after updating the review note. It requires the review to cite the latest passed full harness report, current blockers, objective coverage, and artifact counts.

The `weak-suite-runner` suite dry-runs all cooperative and adversarial weak-model scenarios through `scripts\run-weak-scenario-suite.ps1`. It requires execution-written indexes with `ScenarioId`, `ScenarioKind`, `prompt_hash_sha256`, `runner_recorded`, an allowed `claude-copilot` weak proxy model, transcript paths, and `weak_validation_passed: false`. It also feeds those runner indexes through the live index checker so runner-written ownership cannot drift. The live runner supports quota-friendly resume flags such as `-ScenarioIds`, `-OnlyIncomplete`, `-MaxScenarios`, `-StopOnQuotaBlocked`, and `-NoFailOnRecordedBlocker`. Live uses of the runner keep per-scenario evidence even when hosted quota blocks a model call. Codex-hosted transcripts and Claude bridge aliases that route to Opus are rejected as weak-validation evidence.

`D:\QaaS\_tools\weak-model-session.ps1` re-executes itself under PowerShell 7 when launched from Windows PowerShell 5.1, because live subprocess argument handling needs `ProcessStartInfo.ArgumentList`. Dry-run and live outputs still record `weak_validation_passed: false` unless the live output checker proves the required contract.

The `weak-scenario-outputs` suite classifies the latest live weak-model scenario attempt. It rejects dry-run transcripts as live evidence and validates real weak outputs against per-skill output contracts when quota is available.

The `weak-scenario-live-index` and `weak-adversarial-live-index` suites write scenario-to-transcript index records under `D:\QaaS\_tmp\zappa-dont-cry\blockers`. Each record includes `scenario_id`, `prompt_hash_sha256`, `model`, `dry_run`, `transcript_exit_code`, `match_status`, and the transcript path. If a runner-written index exists in the evidence root, the checker prefers it over transcript rediscovery. The allowed weak proxy model list comes from `D:\QaaS\_tools\weak-model-policy.json` under `claude-copilot`/`airgapped`: `id:gpt-3.5-turbo`, `id:gpt-3.5-turbo-0613`, `gpt-4o-mini`, and `gpt-4o-mini-2024-07-18`. Output classifiers consume these indexes instead of inferring ownership from the newest transcript.

The `airgapped-live` suite classifies the latest live hosted weak-model attempt as either a real live smoke pass or a quota blocker. It rejects dry-run transcripts as live evidence.

The `copilot-fallback` suite classifies the latest hosted Copilot fallback attempt across the configured airgapped fallback models. It records unavailable models and quota-blocked models separately.
